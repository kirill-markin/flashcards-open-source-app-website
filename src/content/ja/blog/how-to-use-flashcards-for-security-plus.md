Security+ の演習セットでは、ポートを1つ取り違えただけで10分無駄になることがあります。概念が難しかったからではありません。389、443、514、636 はどれも何となく見覚えがあり、頭の中では「だいたい分かっている」扱いになって、整理し直す前に次の問題へ進んでしまうからです。たいてい **CompTIA Security+ フラッシュカード** が脇役ではなく、本当に役立つ手段に見えてくるのはこういう瞬間です。

Security+ のデッキがすぐ膨らむのには、単純な理由があります。この試験は、似ていて混同しやすい小さな要素をかなり多く含んでいるからです。

- ポートとプロトコル
- 略語
- コントロールの種類
- 攻撃パターン
- ログの手がかり
- コマンドラインツール
- ポリシーの用語
- クラウドやハイブリッド環境のセキュリティ概念

これらを全部1つの巨大デッキに放り込むと、復習はランダムな雑学クイズのようになります。実際につまずいた箇所からカードを作ると、デッキは初めてちゃんとした役目を果たし始めます。

私が信頼している基準はシンプルです。公式 objectives に載っている内容か、実際の誤答から出てきた内容なら、おそらく入れる価値があります。単に参考書で重要そうに見えただけなら、おそらく入れなくていいです。

2026年の **Security+ フラッシュカード** で目指したい違いは、そこです。

## 2026年5月3日現在、対策すべき試験はまだ SY0-701

CompTIA は現在、Security+ 試験バージョン **V7 / SY0-701** を現行版として掲載しており、開始は **2023年11月7日** です。また CompTIA は、通常は開始から3年ほどでリタイアすると案内しているため、現時点では固定の公開終了日があるというより、見込みは **2026年** です。

ここが重要なのは、古い勉強法の多くが、いまだに SY0-601 時代の習慣や古い教材の山を前提にしているからです。

現在の Security+ ページでは、SY0-701 は実務寄りのセキュリティ業務を軸に説明されています。具体的には、企業のセキュリティ体制の評価、解決策の提案と実装、クラウド・モバイル・IoT を含むハイブリッド環境の保護、ガバナンスとコンプライアンスへの対応、セキュリティインシデントの特定と対応です。何をフラッシュカードにするかも、その前提で決めるべきです。

CompTIA は、現行版の Security+ 試験 objectives と practice questions も公開しています。カードの題材としては、ランダムなデッキ、コピーされた解答集、あるいはダンプ寄りのものより、こちらのほうがずっと信頼できます。

私は、一度触れたあらゆるサイバーセキュリティ用語を展示する博物館のようなデッキは作りません。

プレッシャーの中で素早く引き出す必要がある区別を中心に、デッキを組みます。

この資格を受け続ける人が多いのにも理由があります。米国労働統計局は、information security analysts の雇用が **2024年から2034年にかけて29%増加** すると予測しており、雇用主は専門資格を持つ候補者を好む場合があるとも述べています。これで Security+ がすべての職種で必須になるわけではありませんが、試験対策の需要が消えない理由にはなります。

## SY0-701 では、カードにすべきものが変わる

ありがちな失敗は、Security+ を用語集の書き出しにしてしまうことです。

そうすると、誰も復習したくないカードができます。

- 略語の巨大な羅列
- ポート表の丸ごとコピー
- ガイドから写したポリシー定義
- 1枚の裏面に3つの考え方が入った長い攻撃説明
- 演習問題の解説を丸ごと貼り付けたカード

こういうやり方は、最初の2日くらいはちゃんとして見えます。

そのあと、復習キューがひどいことになります。

SY0-701 なら、私は時間制限つきの想起で崩れやすい部分に寄せてカードを作ります。

- 何度も混ざるポートとプロトコルの組み合わせ
- 見覚えはあるのに意味がぼやけたままの略語
- 攻撃手法とコントロールの区別
- コマンドやツールの認識
- ポリシー、リスク、コントロールの概念どうしの混線
- 演習問題で繰り返し出る誤答

こうして初めて **SY0-701 フラッシュカード** は、試験ガイドの全段落をそのまま写したデッキよりも実用的になります。

## objectives はフィルターとして使い、書き写し作業にはしない

勉強中は公式の objectives 一覧を開いておきますが、箇条書きのすべてをカードに変えようとはしません。

それをやると、一度見覚えができただけで二度と必要にならないような内容ばかりのデッキになりがちです。

もっとよいやり方は、小さく進めることです。

- objective の1セクションを読む
- 自分でうまく説明できなかった用語や区別に印を付ける
- ラボ、演習セット、PBQ で再び出てきたミスだけを追加する

こうすると、各カードに存在理由が生まれます。

Security+ で良いカードの題材になるのは、たいてい次のどれかです。

- 近い概念と混同し続けている用語
- 時間に追われると逆にしてしまうポートとプロトコルの組み合わせ
- 展開はできても使えない略語
- 演習問題や PBQ で見落とした手がかり

これが、**Security+ フラッシュカード** を一般的なサイバーセキュリティ雑学へ逸らさず、試験に結び付け続けるやり方です。

## ポートとプロトコルは、小さな比較カードにする

これは **Security+ ポートとプロトコルのフラッシュカード** が最も機能しやすい領域の1つです。

たいてい失敗の原因は、完全に知らないことではありません。干渉です。似た数字や名前が多すぎて、だいたい同じ記憶の場所を占有してしまいます。

「重要なポートを全部答えよ」という巨大カードは避けます。

その代わり、次のような小さいプロンプトを使います。

- 表: TCP 22 に一般的に対応するプロトコルは?
  裏: SSH。

- 表: 通常、TCP 443 に対応する安全な Web プロトコルはどれ?
  裏: HTTPS。

- 表: 機器ログの集約で UDP 514 を使うことが多いプロトコルは?
  裏: Syslog。

- 表: TCP 636 に一般的に対応するセキュアなディレクトリサービス用プロトコルは?
  裏: LDAPS。

- 表: 実用上、port 53 の DNS と port 443 の HTTPS の違いは何か?
  裏: DNS は名前をアドレスに解決し、HTTPS は TLS で Web トラフィックを保護する。

最後のタイプは、思っている以上に大事です。番号だけのカードも役立ちますが、同じミスを繰り返さないようにするのは、比較カードのほうであることが多いです。

逆向きカードも、何度も落とすポートにだけ使うのが好きです。`LDAPS -> 636` が何度も抜けるなら、`636 -> LDAPS` も独立したカードにする価値があります。まったく落とさないポートまで、完全性のためだけに追加レビューへ押し込む必要はありません。

## 略語は、展開だけでなく意味まで取れるようにする

略語の多い試験では、「見覚えがあるだけ」の状態が痛い失点につながります。文字列を見て少し安心し、選択肢を見た瞬間に、使える意味を作れていなかったことが露呈します。

だから **Security+ 略語フラッシュカード** は、用語集を丸暗記する形で書いてはいけません。

弱いカード:

- 表: SAML は何の略?
  裏: Security Assertion Markup Language。

より良いカード:

- 表: SAML は通常、何をする仕組みかを1文で言うと?
  裏: シングルサインオンのために、当事者間で認証・認可データを受け渡す。

もう1つ良い型があります。

- 表: SSO と MFA の実務的な違いは何か?
  裏: SSO は複数サービスへの重複ログインを減らし、MFA は認証時に追加の本人確認を加える。

もちろん、文字自体が問題なら展開カードも残します。ただ、展開できても意味が分からないなら、そのカードは未完成です。Security+ では、identity、governance、access management 周りの用語でこれが本当によく起きます。見覚えだけは先にできるのに、実際には使える理解にまで届いていないのです。

## 攻撃、防御、コントロールの区別こそ、現実の誤答がよく出る場所

Security+ の問題は、近い概念どうしをよく出します。

- phishing と vishing と smishing
- hashing と encryption
- deterrent、preventive、detective controls
- risk transfer、risk mitigation、risk acceptance
- authorization、authentication、accounting

これらは、復習しやすい大きさでありながら何度も出てくるので、フラッシュカード向きです。この系統の誤答は、たいてい突拍子もない間違いではありません。「ほぼそれを選ぶつもりだった」近い選択肢を選んでしまうのです。

私なら、説明カードではなく区別カードを書きます。

例:

- 表: hashing と encryption の違いは何か?
  裏: hashing は整合性重視の一方向変換で、encryption は適切な鍵で元に戻せる機密性保護。

- 表: security camera は通常、まずどの control type として捉えるべきか?
  裏: detective control。

- 表: risk transfer と risk mitigation を分けるものは何か?
  裏: transfer は保険や契約などで財務的または運用上の影響を他者へ移し、mitigation は発生可能性や影響を直接下げる。

「security controls を詳しく説明せよ」というカードより、こちらのほうがずっとましです。

## コマンドラインの認識カードは、見落とされがちだが効く

Security+ は本格的なシェル試験ではありませんが、コマンドラインの認識はそれでも重要です。概念は知っていても、演習問題でツール名が出た瞬間に止まる人はかなり多いです。

私なら、次の点を押さえるためにカードを使います。

- そのコマンドが大まかに何をするか
- 必要ならどのプラットフォームのものか
- どんな問題文の手がかりでそのツールを思い出すべきか

例:

- 表: ローカルの IP 設定情報を素早く表示する代表的な Windows コマンドは?
  裏: `ipconfig`。

- 表: DNS レコードの照会によく使われるコマンドラインツールは?
  裏: `nslookup` または `dig`。プラットフォームや文脈による。

- 表: どんな種類の問題で、まず `ping` を思い浮かべるべきか?
  裏: 深いパケット解析ではなく、素早い到達性確認。

- 表: `traceroute` と `ping` のフラッシュカード向きの違いは何か?
  裏: `ping` は到達性とレイテンシを確認し、`traceroute` は経路上のホップを示す。

特定のフラグで何度も落ちるのでなければ、すべてのオプションを暗記しようとはしません。Security+ は、英雄的なコマンド雑学よりも、認識と実務的な用途を評価することが多いです。誤答に一度も出てこないコマンドは、デッキで特別扱いする必要はありません。

## PBQ は、誤答のあとに小さなカードへ分解する

CompTIA は Security+ に performance-based questions が含まれると案内しており、Security+ exam FAQ では、その多くが試験の最初に出ると説明しています。だから、この試験でフラッシュカードをどう使うかも変わります。

PBQ 全体を1枚の巨大カードにしようとはしません。

たいてい破綻します。

- 問題文が長すぎる
- 解答も長すぎる
- 採点が曖昧になる
- 復習が遅くなる

有効なのは、繰り返し出る PBQ のミスを、小さいプロンプトに変えることです。

ここで初めて **Security+ PBQ フラッシュカード** が本当に役立ちます。

例:

- 表: firewall や segmentation のシナリオで、ルールを変える前にまず特定すべきものは?
  裏: source、destination、protocol または port、そして意図している業務上またはセキュリティ上の結果。

- 表: log analysis のシナリオでは、どんなものが良いフラッシュカードの題材になるか?
  裏: 繰り返しの認証失敗、impossible travel、異常な outbound traffic、service-port mismatch など、自分が見落とした具体的な手がかり。

- 表: PBQ 由来のフラッシュカードで残すべきものは?
  裏: シミュレーション全体ではなく、繰り返し出る判断ポイントか見落とした手がかり。

PBQ でミスしたあとなら、私はたいてい次の3つを書き出してから先へ進みます。

- 見落とした手がかり
- 判断が遅れた、または間違えたポイント
- その下にある小さなルールや設定パターン

こうすると、デッキは実際の想起に沿い続けます。フラッシュカードだけで PBQ を解けるようにはなりません。ただ、シナリオが速く動き始めたときに、構成要素を見抜きやすくはしてくれます。

## 形のない1つの Security+ デッキを作らない

ここで、多くの資格試験デッキは崩れます。

すべてのカードを `Security+` という大きな1つのバケツに入れてしまうと、毎日の復習は少し雑な同僚にランダムで問題を投げられているような感じになります。

- あるポート
- ある略語
- あるポリシーカード
- ある Linux コマンド
- あるクラウドセキュリティの区別
- 演習解説の残りかすのような曖昧な1枚

私は、まず機能で分け、そのあとトピックでタグ付けします。

たとえば次のようにします。

- deck: `Security+`
- tags: `ports`, `acronyms`, `commands`, `controls`, `pbq-miss`, `logs`, `networking`, `iam`, `compliance`

弱点でタグ付けしてもいいです。

- `still-mixing-up`
- `needs-rewrite`
- `high-value`
- `official-objectives`
- `practice-miss`
- `lab-miss`

こうすると、何でも1つの平たい山に積むより、ずっと使いやすくなります。演習テスト前に短い対象別セッションを回すのも簡単です。

## 毎週のワークフローは、地味なくらいでいい

Security+ のワークフローは、ごく普通の1週間でも続けられるくらい小さく保ちたいです。

1. 公式 objectives、ノート、ラボ教材の1区切りを読む。
2. 短い演習問題セットを解く。
3. 誤答、迷い、際どかった問題だけを、プレーンテキストのメモ1つに記録する。
4. 読んだ内容すべてではなく、そのミスから表裏カードを下書きする。
5. 曖昧なカードはすぐ消す。
6. 残ったカードを FSRS で復習する。

これで十分です。

本1冊の半分を永久復習に流し込むような週末プロジェクトは要りません。

流出問題や exam dumps にも手を出さないほうがいいです。デッキを悪くする理由は2つあります。倫理的に明らかな問題と、実務的な問題です。「傾向を見るだけ」のつもりでも、たいていは公式 objectives、正規の演習教材、自分の誤答メモから本当の想起を作る代わりに、壊れやすい文面の暗記へ寄っていきます。

AI が雑なカード下書きを出してくるなら、次の記事が役立ちます。

## FSRS が Security+ 対策で役立つのは、デッキを絞ったあと

ここで、スケジューラが悪い入力まで救ってくれると期待する人がいます。

でも、それは無理です。

デッキが曖昧な略語カード、巨大な PBQ 要約、重複したポート問題でいっぱいなら、復習システムはその混乱をそのまま処理するしかありません。

FSRS がうまく機能するのは、カードが次の条件を満たすときです。

- 小さい
- 具体的
- 自分で正直に採点しやすい
- 本当に重要な誤答から作られている

これが **CompTIA Security+ フラッシュカード** を持続可能にします。スケジューラがやるのはタイミング管理です。キューに入る資格があるカードを決めることではありません。

## オフライン復習は、資格試験対策で思っている以上に重要

資格試験の勉強は、細切れの時間で進みます。

- 通勤中
- ラボの合間
- 出勤前
- 仕事終わりで、もう頭が疲れている時間
- 待ち時間
- 試験遠征前、ホテルの不安定な Wi-Fi しかない時間

だから私は、この種の試験には offline-first の復習が合うと思っています。ネットが不安定でも、期限カードをちゃんと片付けられるべきです。Security+ 対策は、今いるカフェの回線次第で復習キューが止まっていいほど簡単ではありません。

## このワークフローで Flashcards が合う理由

[Flashcards](https://flashcards-open-source-app.com/) は、**Security+ フラッシュカード** とかなり相性がいいです。5つの別ツールをつなぎ合わせなくても、すでにワークフローに合った形になっているからです。

使えるのは、たとえば次の機能です。

- ワークスペース内データとファイル添付を使える AI チャット
- ノート、objective 一覧、誤答メモを読み込めるプレーンテキストのアップロード
- 段落だらけのカードではなく、表裏カードの作成
- 機能別・弱点別に整理できるデッキとタグ
- カードが整ったあとに回せる FSRS 復習
- Web、iPhone、Android での offline-first 復習
- 技術的に管理したい人向けのオープンソースコードとセルフホスティング

実際には、勉強中の公式 objective の箇条書きを貼り付け、誤答メモやノートのファイルを添付し、AI チャットにもっと絞った表裏カードを下書きしてもらい、編集に耐えたものだけ残す、という流れになります。

もちろん、これで Security+ に魔法のように受かるわけではありません。

ただ、ワークフローとしては筋が通っています。

1. 小さなソース教材を1かたまり集める
2. AI チャットでカードを下書きするか、整える
3. 復習する価値のあるカードだけ残す
4. デッキとタグで整理する
5. 空いた時間にどこでも FSRS で復習する

ポート、略語、演習の誤答を別々の場所に保管して、都合よくまた戻ってくることを祈るより、ずっと筋が通っています。

## 復習に耐える Security+ デッキを作る

良い **SY0-701 フラッシュカード** は、いちばん大きいデッキではありません。

演習問題を2週間続けたあとでも、まだきれいに回せるデッキです。

Security+ で滑りやすい部分にだけ、フラッシュカードを使います。

- ポートとプロトコル
- 実用的な意味まで含めた略語
- コントロールとリスクの区別
- コマンドの認識
- 繰り返し出る PBQ の誤答

何でも保存したくなる誘惑は捨ててください。

公式 objectives を近くに置き、演習問題を正直に使い、繰り返すミスを小さいカードに変え、そのうえでタイミングは FSRS に任せる。

そうすれば、2週目にはもう嫌になっている600枚デッキより、本番で本当に役立つデッキにずっと近づけます。
